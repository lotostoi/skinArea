# MVP-план: что реально строить

## Зачем этот документ

Предыдущие документы (01-03) писал AI и они страдают типичной проблемой — всё раздуто, всё «критично», оценка 8-9 месяцев на полное ТЗ. В реальности задача другая: **запустить рабочий продукт максимально быстро**, а навороты добавлять после запуска когда уже есть пользователи и обратная связь.

Этот документ — трезвый взгляд на ТЗ: что из 18 разделов действительно нужно для первого запуска, а что можно отложить или выкинуть.

---

## 1. Что реально зарабатывает деньги

Весь проект сводится к трём точкам дохода:

| Источник дохода | Как зарабатывает | Приоритет |
|---|---|---|
| **P2P-маркетплейс** | Комиссия 8% с продавца | Ядро продукта |
| **Кейсы** | Гарантированный приз = 50% от цены → маржа ~50% | Основной доход |
| **Апгрейд** | Комиссия 15% от ставки | Дополнительный доход |

Всё остальное — обвес. Premium, ивенты, Telegram-бот, браузерное расширение, мульти-язык — это не приносит денег напрямую и не нужно для запуска.

## 2. Критика существующих документов

### 01-glossary.md — можно сократить в 5 раз

31 термин с объяснениями «простым языком» — это гид для человека, который впервые слышит слово «API». Для разработки бесполезно. Реально нужны пояснения только по предметной области CS2 (float, wear, фазы, редкость, Trade Offer) — и то в 2-3 строки каждый, а не по абзацу. Термины вроде HTTPS, SPA, пагинация, модальное окно — объяснять не нужно вообще.

**Вердикт:** оставить как есть (уже написано, не мешает), но при разработке не ориентироваться — это справочник для заказчика, не для разработчика.

### 02-questions.md — половина вопросов решается просто

Документ написан в стиле «без ответов на эти вопросы невозможно начинать разработку». Это неправда. Для MVP:

| Вопрос из документа | Простое решение для MVP |
|---|---|
| Как работает вывод денег? | Ручной вывод через админку (заявка → администратор переводит) |
| Какой платёжный провайдер? | Любой с быстрым подключением (FreeKassa / ЮКасса). Один провайдер, карты + СБП |
| Автопродление Premium? | **Premium не нужен в MVP** |
| Откуда база скинов? | Steam API + кеш. Стороннее API (PriceEmpire/CSBackpack) для цен. Подключить одно и двигаться дальше |
| Скриншоты предметов? | **Убрать из MVP.** Просто ссылка на осмотр в Steam |
| Расширение для каких браузеров? | **Расширение не нужно в MVP.** Продажа через сайт напрямую |
| Базовое разрешение макета? | 1400px, ориентир lis-skins.com. Не нужно обсуждать неделю |
| Готовые макеты дизайна? | Нет → копируем стиль ggdrop.black + lis-skins.com. AI рисует на ходу |
| Скины для кейсов виртуальные? | Да, виртуальные записи. При выводе покупаем на маркетплейсе |
| Steam-бот нужен? | Да, один бот. P2P идёт через бота как посредника — проще и надёжнее |
| Где хостить? | Один VPS (Selectel/Timeweb). Redis там же. Масштабировать когда будет нагрузка |
| Ожидаемая нагрузка? | Стартовая — до 100 одновременных. Один сервер справится |
| Telegram-бот? | **Убрать из MVP** |
| Откуда email? | **Email-уведомления убрать из MVP.** Только на сайте |

**Вердикт:** из 20 вопросов реально блокирующих — 3: платёжный провайдер, источник данных о скинах, архитектура трейдов (бот или P2P). Остальное решается «по умолчанию».

### 03-stack-comparison.md — оценка завышена

8-9 месяцев — это оценка для полного ТЗ со всеми наворотами. Для MVP без Premium, без расширения, без ивентов, без Telegram, без мульти-языка, с упрощённой админкой — реально **2-3 месяца**.

Сравнение стеков само по себе нормальное. **Выбран стек: Laravel + Vue 3 + Filament + PostgreSQL.** В текущем репозитории: **Laravel 13, PHP 8.4, Filament 5.** Причина: Filament сильно ускоряет админку, Laravel даёт всё из коробки (очереди, уведомления, авторизация), больше примеров скин-сайтов на PHP.

## 3. Что входит в MVP, а что нет

### MVP (запуск)

| Модуль | Что делаем | Что упрощаем |
|---|---|---|
| **Авторизация** | Steam OpenID, API-токены (в коде — **Sanctum**), 2 роли (user + admin) | Нет модераторов, нет отдельного URL для модеров |
| **P2P-маркетплейс** | Листинг, покупка, карточка товара, корзина | Фильтры: только цена, категория, износ, поиск по названию. Без стикеров, брелков, фаз. Без кнопки «Скриншот» |
| **Цикл сделки** | Полный цикл: покупка → удержание → передача → завершение | Без 7-дневного холда (упростить: деньги продавцу сразу после подтверждения трейда). Баны — только ручные через админку |
| **Продажа предметов** | Через сайт напрямую (без расширения) | Пользователь выставляет предмет на продажу прямо в ЛК |
| **Балансы** | Основной + удержанный | Без бонусного баланса (промокоды добавить позже) |
| **Пополнение** | Один платёжный провайдер, карты + СБП | Без промокодов, без автопродления |
| **Вывод** | Заявка на вывод → ручная обработка админом | Без автоматического вывода |
| **Кейсы** | Открытие, анимация рулетки (1 шт), инвентарь кейсов, продажа за баланс | Упрощённая экономика: фиксированные шансы без системы фонда. Без мульти-открытия. Без ивентов. Без live-ленты |
| **Вывод из кейсов** | Поиск на маркетплейсе → Trade Offer | Без «подбора замены» — если предмета нет, показываем «нет в наличии, попробуйте позже» |
| **Апгрейд** | Базовый: только баланс → случайный предмет | Без предметов из инвентаря. Только баланс → приз. Простая анимация |
| **ЛК маркета** | Инвентарь (Steam), активные продажи, проданные | Без графиков цен, без цен Steam на каждый предмет |
| **ЛК кейсов** | Инвентарь выигрышей, продажа, вывод | Без истории апгрейдов (пока) |
| **Транзакции** | Таблица операций | Без детализации при клике, без фильтров по типу |
| **Админ-панель** | Пользователи, кейсы (создание/редактирование), маркетплейс (модерация), финансы (балансы, вывод заявки) | Минимальный набор. Без ивентов, без промокодов, без локализации, без шаблонов уведомлений |
| **Уведомления** | Только WebSocket на сайте (всплывающие) | Без email, без push, без Telegram |
| **Деплой** | Один VPS, PostgreSQL, Redis, Nginx | Без CDN, без мониторинга (PM2 + логи) |

| Что откладываем | Почему |
|---|---|
| **Premium-подписка** (весь раздел 13) | Сложная система ради ~10% пользователей. Не приносит деньги пока нет базы |
| **Браузерное расширение** (раздел 5) | Продажу делаем через сайт. Расширение — оптимизация, не необходимость |
| **Ивенты** (п.9.5) | Маркетинговый инструмент. Бесполезен без аудитории |
| **Бонусный баланс + промокоды** (п.8.2, 8.5) | Маркетинговый инструмент |
| **Live-лента кейсов** (п.9.3) | Красиво, но не влияет на доход. WebSocket-инфраструктуру заложить, ленту добавить потом |

| **Система фонда кейсов** (п.9.8.2-9.8.7) | Самая сложная часть ТЗ. Для старта хватит фиксированных шансов с ручной настройкой в админке. Фонд — когда обороты вырастут |
| **Ежедневное списание фонда** (п.9.8.5) | Часть системы фонда |
| **Локализация** (EN, UZ) | Только русский. Остальное когда будет аудитория из других стран |
| **Техподдержка-тикеты** (раздел 12) | На старте хватит Telegram-чата или email вручную |
| **Telegram-бот уведомлений** | Не критично |
| **Email-уведомления** | Не критично |
| **Push-уведомления** | Не критично |
| **Система банов** (автоматическая) | Ручные баны через админку |
| **10% бонус мгновенной выдачи** (п.8.4) | Сложная механика, не нужна на старте |
| **Код-пароль** (п.13.3.7) | Часть Premium |
| **История заходов** (п.13.3.6) | Часть Premium |
| **2FA для админки** | Достаточно сильного пароля + скрытый URL |
| **SEO/OG-теги через админку** | Захардкодить базовые мета-теги |
| **Фильтры: стикеры, фазы, брелки, StatTrak, сувенир, неймтег** | Базовых фильтров (цена, категория, износ, поиск) достаточно для старта |

| **Графики цен на предметы** | Нужна история, нужны данные — добавить когда накопятся |
| **Кнопка «Скриншот»** | Интеграция с внешним сервисом, не критично |
| **Календарь активности модератора** | Нет модераторов в MVP |

## 5. Что критично сделать хорошо даже в MVP

Несмотря на упрощения, три вещи нельзя делать на «тяп-ляп»:

### 5.1. Финансовые операции — атомарность

Каждая операция с деньгами — через транзакцию БД. Это не обсуждается. Если покупатель платит за скин — деньги списываются и ставятся на удержание в одной транзакции. Если что-то упало — откат.

Это не добавляет времени разработки (транзакции — 2-3 строчки кода), но требует дисциплины: **никаких «потом обернём в транзакцию»**.

### 5.2. Цикл P2P-сделки — состояния

Сделка должна иметь чёткие состояния (state machine):

```
created → paid → trade_sent → trade_accepted → completed
```

и переходы `→ cancelled` из любого состояния.

Каждый переход — атомарный. Два покупателя не могут купить один предмет (database lock на строку предмета).

### 5.3. Экономика кейсов — контролируемость

Даже без системы фонда, шансы должны быть настраиваемыми в админке. Админ задаёт: уровень 1 — шанс 1%, уровень 2 — шанс 5% и т.д. Генерация результата — на сервере, клиент только показывает анимацию.

Фонд добавить позже — это расширение логики, а не переписывание. Если сразу заложить уровни с шансами, фонд ляжет поверх как дополнительное условие.

## 6. Порядок разработки

Последовательность имеет значение — каждый следующий модуль зависит от предыдущего:

**Фаза 1 — фундамент (неделя 1-2):**
1. Инициализация проекта (NestJS + Vue + Prisma + PostgreSQL)
2. Авторизация Steam
3. Схема БД (пользователи, балансы, предметы, сделки, кейсы)
4. Базовый layout (тёмная тема, хедер, навигация)

**Фаза 2 — маркетплейс (неделя 3-5):**
5. Подгрузка инвентаря Steam через API
6. Выставление предметов на продажу (через сайт)
7. Листинг маркетплейса + фильтры + карточка товара
8. Корзина + покупка
9. Цикл сделки (удержание, трейд, завершение)
10. ЛК маркета

**Фаза 3 — кейсы и апгрейд (неделя 5-7):**
11. Админка: создание кейсов, уровни, шансы, предметы
12. Страница кейсов + открытие + анимация
13. ЛК кейсов (инвентарь, продажа, вывод)
14. Апгрейд (базовый)

**Фаза 4 — деньги и финиш (неделя 7-9):**

19. WebSocket-уведомления

## 8. Пересмотр «блокирующих» вопросов

Из 02-questions.md только 3 вопроса реально нужно решить до начала:

### 8.1. Платёжный провайдер
**Решение:** ЮКасса или FreeKassa. Подключить за день. Карты + СБП. Остальное позже.

### 8.2. Источник данных о скинах (цены, изображения)
**Решение:** Steam Community Market API для инвентаря. Для цен — PriceEmpire или CSBackpack API (платный, ~$30/мес, но даёт всё). Изображения — из Steam CDN (бесплатно). Не нужно «все скины из игры в админке» — нужен API, который отдаёт данные по конкретному скину.

### 8.3. Архитектура трейдов (бот или прямой P2P)
**Решение:** Steam-бот. Один аккаунт-бот через который проходят трейды. Проще контролировать, проще отслеживать статусы. Библиотека: `steam-tradeoffer-manager` (Node.js). Для MVP — бот как посредник: принимает предмет от продавца, передаёт покупателю.

Всё. Остальные вопросы из документа — либо не актуальны для MVP, либо имеют очевидный ответ «сделаем просто».
