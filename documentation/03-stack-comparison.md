# Сравнение стеков: Laravel + Vue vs Node.js + Vue

## О чем этот документ

Это сравнение двух вариантов технологического стека для проекта SkinsArena:

- **Вариант A:** Laravel (PHP) + Vue.js
- **Вариант B:** Node.js (Express/NestJS + TypeScript) + Vue.js

Фронтенд (Vue.js) одинаковый в обоих вариантах — разница только в бэкенде.

Важный контекст: код будет писаться преимущественно с помощью AI (Cursor), а человек выступает в роли архитектора — задает направление, проверяет результат, принимает решения. Это существенно влияет на оценку.

В конце — конкретная оценка трудозатрат по каждому модулю проекта и рекомендация.

### 1.2. Подробный разбор ключевых моментов

#### WebSocket и реальное время

Это один из самых важных критериев для SkinsArena. Проект интенсивно использует реальное время:

- Live-лента кейсов (обновляется у всех пользователей мгновенно)
- Уведомления продавцу о покупке (со звуком)
- Обновление статусов сделок
- Анимации открытия кейсов (результат с сервера)

**Laravel:** PHP по своей природе "умирает" после каждого запроса — обработал и забыл. Для WebSocket нужен отдельный сервис (Soketi, Pusher, или laravel-websockets). Laravel Echo на клиенте + Soketi на сервере — работает, но это дополнительный компонент, который нужно настраивать, поддерживать и масштабировать отдельно от основного приложения.

**Node.js:** JavaScript работает в событийной модели — один процесс постоянно живет и обрабатывает события. WebSocket через Socket.io — это естественная среда для Node. Нет дополнительных сервисов, все в одном приложении. Логика "при покупке отправить уведомление продавцу" пишется прямо в обработчике покупки.

**Вывод:** Для проекта с интенсивным реальным временем Node.js имеет существенное преимущество. Это не значит, что Laravel не может — может, но через дополнительную инфраструктуру.

#### Админ-панель

SkinsArena имеет масштабную админ-панель (раздел 15 ТЗ): управление пользователями, кейсами, маркетплейсом, ивентами, подписками, уведомлениями, финансами, локализацией.

**Laravel:** Filament PHP — генерирует полноценную админку из описания моделей. Таблицы, формы, фильтры, статистика — за часы, не за недели. Это один из главных козырей Laravel. Для SkinsArena с 12+ разделами админки — это может сэкономить несколько недель работы.

**Node.js:** AdminJS — существует, но значительно менее зрелый. Чаще всего админку пишут вручную на React/Vue. Это больше времени, но больше контроля над внешним видом и поведением.

**Вывод:** Laravel сильно побеждает в части админки. Для проекта с такой обширной админ-панелью это серьезный аргумент.

#### Один язык на фронте и бэке

**Node.js:** TypeScript на фронте (Vue) и бэке (Node) — это один язык. Типы можно шарить между фронтом и бэком (например, типы ответов API, типы данных скинов). Это уменьшает количество ошибок и ускоряет разработку.

**Laravel:** PHP на бэке, TypeScript на фронте — два разных языка с разной типизацией. Нужно поддерживать типы в двух местах. При изменении структуры данных нужно менять в обоих.

**Вывод:** Для AI-разработки в Cursor единый язык — преимущество: AI видит весь контекст проекта в одном языке и лучше понимает связи.

### 2.2. Детальное обоснование оценки по каждому модулю

Ниже — пункт за пунктом: что конкретно нужно сделать, почему именно столько, какую долю работы берёт на себя AI, а что ложится на архитектора. Оценки даны для Node.js + Vue (для Laravel отличия указаны отдельно, где есть).

#### 1. Авторизация (Steam OpenID) — 1 неделя

**Что нужно сделать:**
- Интеграция Steam OpenID (редирект → получение Steam ID → создание/обновление пользователя)
- JWT-токены (выдача, обновление, отзыв)
- Роли и права (пользователь, модератор, администратор) — guards/middleware
- Обработка кастомных URL Steam-профилей (`/id/customname/` и `/profiles/7656...`)

**Почему 1 неделя:**
Стандартная задача — passport-steam для Node, пакеты для Laravel. Основное время уходит не на саму авторизацию (2 дня), а на систему ролей и прав, которая пронизывает весь проект (3 дня). Нужно заложить архитектуру прав, которая потом не будет переписываться.

**AI: помогает хорошо (70%).** Steam OpenID — типовая интеграция, тысячи примеров. JWT — стандарт. Роли — стандартный паттерн. Архитектор задаёт структуру, AI генерирует код.

---

#### 2. P2P-маркетплейс: листинг, покупка, фильтры — 3 недели

**Что нужно сделать:**
- Страница маркетплейса: сетка карточек предметов с тенями редкости, float-цветами, hover-эффектами (ТЗ п.7.7)
- 12 категорий оружия (ножи, перчатки, пистолеты...) — табы сверху
- Система фильтров слева: цена, раритетность, износ, float (от/до), фаза, стикеры, брелок, StatTrak, сувенир, неймтег (ТЗ п.7.6)
- Карточка товара — детальное модальное окно: крупное изображение, float, кнопка «Осмотр в игре», кнопка «Скриншот», блок «Похожие предметы» с разделением по износу (ТЗ п.7.7.3)
- Корзина — модальное окно с выбранными предметами, общей суммой, кнопкой «Оплатить» (ТЗ п.7.8)
- Пагинация (10/20/50/100/500)
- API для всего этого: листинг с фильтрацией, сортировкой, пагинацией
- Подгрузка данных о стикерах и раритетности из Steam (ТЗ п.7.6.2)

**Почему 3 недели:**
Объём фронтенда большой — одних фильтров 12 штук, причём некоторые динамические (стикеры подгружаются из Steam). Карточка товара сложная — тень редкости, float-цвет, hover-анимация, детальное окно с «похожими предметами». Бэкенд: фильтрация по множеству параметров с пагинацией и производительностью. 1.5 недели фронтенд + 1 неделя бэкенд + 0.5 недели интеграция и отладка.

**AI: помогает хорошо для CRUD (70%), средне для фильтров (40%).** Карточки, пагинацию, модальные окна AI напишет быстро. Но: тонкая настройка теней редкости (7 разных цветов), правильная работа множественных фильтров одновременно, подгрузка стикеров из Steam — тут AI потребуется много контекста и ручная доводка.

#### 3. P2P: цикл сделки — 2.5 недели (Node) / 3 недели (Laravel) — **дорогой модуль**

**Что нужно сделать:**
- 5 этапов сделки: создание продажи → покупка → передача предмета → получение → полное завершение (ТЗ п.7.2)
- Списание с баланса покупателя → удержание у продавца → проверка Steam Trade → разблокировка
- Steam Trade Offer: создание, отслеживание статуса, отмена
- Таймауты: 15 минут на подтверждение трейда, 7 дней Steam Protection (ТЗ п.7.3)
- Автоматические отмены при истечении таймаутов (cron/очереди)
- WebSocket-уведомления продавцу (звук + всплывающее окно) при каждом изменении статуса
- Система банов: 3 раза не передал = бан на час, отменил трейд = бан на 3 месяца (ТЗ п.7.5)
- Учёт 7-дневной Steam Protection — деньги на удержании до окончания холда (ТЗ п.7.4)
- 10% бонус мгновенной выдачи — начисление, использование при следующей продаже, стакание, срок действия 30 дней (ТЗ п.8.4)

**Почему 2.5-3 недели — это САМЫЙ ДОРОГОЙ модуль по сложности:**
Это ядро бизнес-логики с максимальной плотностью edge-cases. Каждый из 5 этапов сделки может пойти не так: продавец не передал, покупатель не принял, Steam-трейд отменён, таймаут истёк, у продавца трейд-бан появился после создания продажи. Все финансовые операции атомарные — если сервер упал между списанием у покупателя и начислением удержания продавцу, система не должна потерять деньги. Race condition: два покупателя одновременно покупают один предмет — нужны блокировки. Таймауты через очереди (delayed jobs) с гарантией исполнения. Node чуть быстрее (0.5 нед) из-за нативного WebSocket и удобных Steam-библиотек.

**AI: помогает слабо (15-20%).** AI может написать базовую структуру (контроллеры, сервисы, очереди), но: логику состояний сделки, атомарные транзакции, race condition protection, правильную обработку таймаутов — всё это проектирует и проверяет архитектор лично. Ошибка тут = потеря реальных денег.

#### 4. Система балансов (3 типа, удержание) — 2 недели — **дорогой модуль**

**Что нужно сделать:**
- Основной баланс: пополнение (деньги, продажа предметов, продажа из кейсов), списание (покупка, кейсы, апгрейд, вывод) (ТЗ п.8.1)
- Бонусный баланс: только из промокодов, только на кейсы, приоритет списания бонусный → основной (ТЗ п.8.2)
- Удержанный баланс: заморозка при P2P, разблокировка после завершения сделки, возврат при отмене (ТЗ п.8.3)
- Промокоды: валидация, активация при пополнении, начисление бонусов (ТЗ п.8.5)
- Отображение всех трёх балансов в ЛК с кликабельным «удержанным» (модальное окно с транзакциями)
- Единая таблица транзакций с типами: пополнение, покупка, продажа, кейс, апгрейд, вывод (ТЗ п.6.7)
- Детализация каждой транзакции (разворачивающийся блок с изображением предмета)

**Почему 2 недели:**
Три типа балансов с разными правилами — это три разных «кошелька» с ограничениями. Приоритет списания (бонусный → основной) при открытии кейсов — нетривиально, когда бонусных не хватает и нужно частично списать с обоих. Удержанный баланс привязан к состояниям P2P-сделок — нужна строгая связь. Промокоды с лимитами (на пользователя, общий, по дате). Каждая операция с балансом — транзакция БД + запись в лог. 1 неделя бэкенд (модели, сервисы, транзакции) + 0.5 недели API + 0.5 недели фронтенд (отображение, таблица транзакций).

**AI: помогает слабо (20%).** AI сгенерирует модели и CRUD транзакций, но: атомарность операций, правильный приоритет списания, корректная работа удержания — это ручное проектирование. Одна ошибка = деньги появляются из воздуха или пропадают.

#### 5. Кейсы: интерфейс + анимации — 2 недели

**Что нужно сделать:**
- Страница кейсов: категории (сворачиваемые), сетка кейсов с картинками/ценами/тенями (ТЗ п.9.2)
- Live-лента: движущаяся строка с выпавшими предметами, режимы «ТОП»/«ВСЕ», остановка при наведении (ТЗ п.9.3)
- Открытие кейса: анимация рулетки (одиночное), переворот карточек (мульти), кнопки 1/2/3/4/5/10, «быстро» (ТЗ п.9.4)
- Баннер ивента с таймером, таблица лидеров (ТЗ п.9.5)
- Фильтр над кейсами: цена, поиск, доступность по балансу (ТЗ п.9.7)
- Звуковые эффекты для рулетки
- WebSocket: отправка результата открытия в Live-ленту всем пользователям

**Почему 2 недели:**
Чисто фронтендная задача (бэкенд логики кейсов — отдельный пункт). Анимация рулетки — самая трудоёмкая часть: нужно, чтобы предметы «крутились» плавно, останавливались с замедлением, звук совпадал с визуалом. Мульти-открытие (переворот 10 карточек) — отдельная анимация. Live-лента с WebSocket-подпиской и остановкой при наведении. Ивент-баннер с обратным отсчётом. 1.5 недели верстка и анимации + 0.5 недели WebSocket-интеграция и звук.

**AI: помогает хорошо (60%).** Базовую верстку, компоненты кейсов, WebSocket-подписку AI напишет быстро. Но: тонкая настройка анимации рулетки (тайминги, easing-функции, чтобы «ощущалось» правильно), синхронизация звука — это ручная подгонка, AI не чувствует UX.

#### 6. Кейсы: экономика (фонд, уровни, вероятности) — 2.5 недели — **самый дорогой модуль**

**Что нужно сделать:**
- Нераспределённый призовой фонд для каждой категории кейсов: Потрачено − Выдано (ТЗ п.9.8.2.2)
- 5 уровней призов с настраиваемыми вероятностями и ценами (ТЗ п.9.8.1.2)
- Гарантированный приз = 50% от цены кейса (ТЗ п.9.8.2.1)
- Условие: приз уровня N доступен только если фонд ≥ 2 × стоимость приза (ТЗ п.9.8.3.1)
- Пересчёт вероятностей при одновременной доступности нескольких уровней (ТЗ п.9.8.3.4)
- Алгоритм выдачи: определение доступных уровней → розыгрыш → выбор скина → обновление фонда → логирование (ТЗ п.9.8.4)
- Ежедневное списание фонда 5% (cron, настраиваемое время и процент) (ТЗ п.9.8.5)
- Атомарность: при одновременных открытиях фонд не уходит в минус, приз не выдаётся дважды (ТЗ п.9.8.7.2)
- Списание с баланса (бонусный → основной) при открытии
- Premium-скидки на кейсы (10% до 500₽, 5% дороже 500₽) — ТЗ п.13.3.1
- Анти-анлак (Premium): 10 неокупов подряд в одном кейсе → 11-й бесплатно (ТЗ п.13.3.5)
- Бесплатные кейсы (ограничение на пользователя, от суммы пополнения) — ТЗ п.9.8.1.1

**Почему 2.5 недели — это САМЫЙ СЛОЖНЫЙ модуль:**
Уникальная бизнес-логика, которой нет ни в одном открытом проекте. Пересчёт вероятностей при множественной доступности уровней — математика, которую нужно спроектировать и покрыть тестами (десятки edge-cases). Атомарность: 100 пользователей одновременно открывают один кейс — фонд не должен уйти в минус, каждый скин выдаётся один раз. Ежедневное списание — cron-задача с побочными эффектами (уровни становятся недоступными). Анти-анлак — отдельная система трекинга последних 10 открытий с учётом ID кейса. 1.5 недели бэкенд (алгоритм, транзакции, cron) + 0.5 недели тесты + 0.5 недели интеграция и отладка.

**AI: помогает слабо (10-15%).** Это полностью уникальная логика — AI не видел ничего подобного в обучающих данных. AI может написать скелет сервиса и тесты по спецификации, но: формулы пересчёта вероятностей, корректность условий выдачи, атомарность при конкурентных запросах — всё проектируется вручную. Каждая ошибка = сайт раздаёт дорогие скины бесплатно или наоборот зажимает их навсегда.

Зависит от провайдера, но в среднем: 0.5 недели интеграция пополнения + 0.5 недели обработка вебхуков + 0.5 недели рекуррентные платежи + 0.5 недели вывод средств + фронтенд.

**AI: помогает средне (40%).** AI знает популярные провайдеры (Stripe, ЮКасса), но: каждый провайдер уникален, API документация у российских провайдеров часто неполная. Рекуррентные платежи — сложная интеграция.

---

#### 19. Деплой и инфраструктура — 0.5 недели (Node) / 1 неделя (Laravel)

**Что нужно сделать:**
- Настройка сервера (Nginx, SSL, PM2/PHP-FPM)
- PostgreSQL, Redis — установка и конфигурация
- CI/CD (автодеплой при пуше)
- Мониторинг (PM2 / Supervisor)
- Бэкапы БД

**Почему 0.5-1 неделя:**
Node проще — один процесс PM2 + Nginx. Laravel — PHP-FPM + Nginx + Supervisor для очередей + Soketi для WebSocket = больше компонентов. В обоих случаях: Docker или прямая установка, настройка CI/CD (GitHub Actions).

---

#### 20. Тестирование и отладка — 3 недели

**Что нужно сделать:**
- Юнит-тесты: формулы (шансы кейсов, апгрейд, комиссии), балансы, вероятности
- Интеграционные тесты: полный цикл P2P-сделки, открытие кейса с проверкой фонда
- E2E-тесты: основные пользовательские сценарии
- Нагрузочное тестирование: одновременные открытия кейсов, одновременные покупки
- Ручное тестирование: анимации, звуки, UX

**Почему 3 недели:**
Тесты разбросаны по всему проекту. Самое критичное и долгое: интеграционные тесты финансовых операций (атомарность, race conditions) и экономики кейсов (корректность выдачи, фонд не уходит в минус). Нагрузочное тестирование — нужно эмулировать 100+ одновременных пользователей. 1 неделя юнит-тесты + 1 неделя интеграционные + 1 неделя E2E и ручное.

**AI: помогает хорошо для юнит-тестов (70%), слабо для интеграционных (20%).** AI отлично пишет юнит-тесты по спецификации. Но: тестирование race conditions, сценарии «что если сервер упал между шагами» — архитектор проектирует сценарии, AI помогает реализовать.

## 4. Аргументация: что выбрать

### 4.1. Аргументы ЗА Laravel + Vue

1. **Админ-панель за дни, а не недели.** Filament PHP — это готовая система. Для 12 разделов админки SkinsArena это экономия 2-3 недели. Админку писать нужно и нужно много.
2. **"Все из коробки".** Авторизация (Sanctum), очереди (Queue), уведомления (Notifications), события (Events), миграции — все встроено и работает вместе. Не нужно собирать "конструктор" из npm-пакетов.
3. **Больше примеров для скин-сайтов.** Исторически многие скин-сайты написаны на Laravel (market.csgo.com, skin.club и другие). Больше готовых решений для Steam OpenID и Steam API на PHP.
4. **Проще найти разработчика.** Если в будущем нужно будет подключить другого человека — Laravel-разработчиков на рынке больше (особенно в СНГ).
5. **Структурированность.** Laravel навязывает определенную структуру проекта (MVC). Это плюс для AI — он лучше понимает, куда что положить.

### 4.2. Аргументы ЗА Node.js + Vue

1. **Один язык (TypeScript) везде.** Фронтенд и бэкенд на одном языке. Типы шарятся. AI в Cursor видит весь проект в едином контексте. Меньше переключений, меньше ошибок на стыке.
2. **WebSocket как родная среда.** Для проекта с live-лентой, мгновенными уведомлениями и анимациями кейсов — WebSocket критичен. Node.js обрабатывает тысячи постоянных соединений одним процессом. Laravel требует отдельного сервиса.
3. **Производительность при высокой нагрузке.** Node.js не перезапускает контекст на каждый запрос (в отличие от PHP). Для множества одновременных WebSocket-соединений это важно.
4. **Современный стек.** TypeScript, ESM-модули, нативная асинхронность (async/await без надстроек). Код выглядит чище и предсказуемее.
5. **Проще деплой.** Node-процесс + PM2 + Nginx. Нет необходимости в PHP-FPM, OPcache, Supervisor для очередей отдельно.
6. **Лучше для микросервисной архитектуры в будущем.** Если проект вырастет — выделить WebSocket-сервер, очередь кейсов, платежный сервис проще на Node.
7. **Экосистема npm.** 2.5 миллиона пакетов. Для любой задачи есть готовое решение. Для Steam — node-steam-user, steam-tradeoffer-manager — хорошо поддерживаемые библиотеки.

### 4.3. Аргументы, специфичные для AI-разработки (Cursor)


| Фактор                 | Laravel                                | Node.js                                     | Что это значит                                             |
| ---------------------- | -------------------------------------- | ------------------------------------------- | ---------------------------------------------------------- |
| **Единый контекст**    | PHP + TypeScript (два языка)           | TypeScript + TypeScript                     | AI лучше связывает фронт и бэк, когда все на одном языке   |
| **Типизация**          | PHPDoc аннотации (неполные)            | TypeScript строгий (полная типизация)       | AI делает меньше ошибок с полной типизацией                |
| **Шаблонность**        | Высокая (Laravel навязывает структуру) | Средняя (NestJS навязывает, Express — нет)  | С NestJS — паритет. С Express — AI нужны четкие инструкции |
| **Качество генерации** | Отличное (много примеров)              | Отличное (много примеров)                   | Оба стека хорошо представлены                              |
| **Скорость итераций**  | Быстрая (artisan-команды)              | Быстрая (TypeScript + hot reload)           | Паритет                                                    |
| **Отладка AI-кода**    | PHPStan / Psalm ловят ошибки           | TypeScript ловит ошибки на этапе компиляции | TypeScript строже — ошибки видны раньше                    |


**Ключевой аргумент для AI-разработки:** TypeScript дает строгую типизацию, которая работает как "автоматический ревьюер" — если AI написал код с неправильным типом, ошибка видна сразу, до запуска. В PHP ошибки типов часто всплывают только при выполнении.

## 5. Честное сравнение рисков


| Риск                                              | Laravel                               | Node.js                                                                                  |
| ------------------------------------------------- | ------------------------------------- | ---------------------------------------------------------------------------------------- |
| **"Callback hell" / сложность асинхронного кода** | Нет (PHP синхронный)                  | Есть, но async/await решает. AI иногда забывает await — ошибка тихая                     |

| **Проблемы на проде**                             | PHP проверен годами на хостингах      | Node.js может "утечь" по памяти при долгой работе — нужен мониторинг                     |
| **Отладка финансовых ошибок**                     | Логи понятные, стектрейс PHP читаемый | Асинхронные стектрейсы менее информативны                                                |

## 6. Рекомендация

### Если выбирать объективно для ЭТОГО проекта:

**Node.js (NestJS) + Vue + TypeScript — лучше подходит для SkinsArena.** 

Причины:

1. **WebSocket — основа проекта.** Live-лента, уведомления, статусы сделок — это не "приятное дополнение", а ядро продукта. Node.js обрабатывает это естественнее.
2. **TypeScript на фронте и бэке** — единый язык критичен при AI-разработке. AI в Cursor понимает весь проект целиком, типы можно переиспользовать, ошибки ловятся раньше.
3. **Steam-библиотеки на Node** (node-steam-user, steam-tradeoffer-manager) — наиболее поддерживаемые и полные. Большинство Steam-ботов написаны на Node.
4. **NestJS** дает такую же структурированность, как Laravel (модули, сервисы, контроллеры, middleware, guards). AI одинаково хорошо работает с обоими.
5. **Админ-панель** — да, Laravel выигрывает с Filament, но это 2-3 недели разницы, которые компенсируются преимуществами Node в остальных частях проекта. Кроме того, для кастомной админки с нестандартными требованиями (экономика кейсов, фонды, управление уровнями) Filament все равно придется дорабатывать.

### Если выбирать Laravel:

Разумный выбор, если:

- Уже есть опыт команды с Laravel
- Админка — самая приоритетная часть (хочется запуститься быстро)
- Нет желания разбираться с экосистемой Node
- Нужно быстро найти дополнительного разработчика

### Конкретная рекомендация для ситуации "архитектор + AI":

**Node.js + NestJS + Vue + TypeScript** — потому что:

- AI делает меньше ошибок на TypeScript (строгая типизация как страховка)
- Один язык = один контекст = AI лучше понимает связи в проекте
- WebSocket без дополнительной инфраструктуры = меньше конфигурации = меньше мест, где что-то может сломаться
- Архитектор фокусируется на бизнес-логике (экономика, безопасность), а AI хорошо генерирует типовой код на TypeScript

## Выводы

1. **Оба стека способны реализовать SkinsArena.** Разница в трудозатратах — около 1-2 недель (Laravel быстрее на админке, Node быстрее на WebSocket и деплое).
2. **Для AI-разработки Node.js + TypeScript предпочтительнее:** единый язык, строгая типизация, лучше поддержка WebSocket — это уменьшает количество ошибок, которые AI может допустить.
3. **Самые сложные части проекта одинаково сложны на обоих стеках:** экономика кейсов, финансовые операции, P2P-цикл — тут не фреймворк определяет сложность, а бизнес-логика. AI мало помогает в этих местах независимо от стека.
4. **Laravel — разумный выбор, если важна скорость запуска** (Filament для админки) и уже есть опыт с PHP. Node.js — если важна производительность, единый язык и нативный WebSocket.
5. **Общая оценка: ~8-9 месяцев** при полной загрузке одного архитектора с AI. Это с учетом того, что AI ускоряет типовые задачи на 60-80%, но сложные (финансы, экономика) — лишь на 10-20%.
