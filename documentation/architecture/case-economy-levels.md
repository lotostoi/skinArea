# Экономика уровней кейса: пороговая цена и фонд

Краткая выжимка для админки и кода. Полное описание и примеры — в `documentation/00-technical-specification.md`, раздел **9.8** (подпункты 9.8.2.1, 9.8.3).

## Поле «Пороговая цена уровня» (`prize_amount`)

В БД и коде одно поле используется **в двух разных правилах**. В интерфейсе Filament оно подписано как пороговая цена уровня в рублях.

### 1. Гарантированный уровень (в коде: строка с **наибольшим номером** в поле `level`)

- При **сохранении активного кейса** валидатор требует: `prize_amount` этого уровня **ровно 50%** от **цены открытия** кейса (`GameCase::price`).
- Это соответствует ТЗ **9.8.2.1** (в ТЗ фигурирует «5-й уровень»; в приложении гарантия привязана к **максимальному** `level`, обычно это и есть 5).

Сообщение об ошибке при сохранении относится **только к этому** правилу.

### 2. Не гарантированные уровни (все остальные номера)

- **В момент открытия** уровень попадает в пул призов только если **нераспределённый фонд** кейса **≥ 2 × `prize_amount`** этого уровня.
- Гарантированный уровень **не** проверяется по фонду — он в пуле всегда.

Реализация: `App\Services\CasePrizeFundService::getAvailableLevels`, `CaseEconomyValidator` (только сумма шансов 100%, 50% у гарантированного и активность).

### Зачем в подсказках упоминают «среднюю цену скинов»

Для дорогих (не гарантированных) уровней разумно задавать `prize_amount` близко к типичной стоимости призов в уровне, чтобы правило **2×** отражало реальную нагрузку на фонд. Это **рекомендация**, не отдельная проверка в коде.

## Где смотреть в коде

| Правило | Класс / метод |
|--------|----------------|
| 50% у гарантированного при сохранении активного кейса | `CaseEconomyValidator::validate` |
| Фонд ≥ 2× для остальных уровней при открытии | `CasePrizeFundService::getAvailableLevels` |
| Баланс фонда | `CasePrizeFundService::getFundBalance` |
