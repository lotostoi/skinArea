# Контрактные решения рефакторинга

## Цель

Зафиксировать решения по спорным зонам контракта, чтобы backend, frontend и тесты работали согласованно.

## Решение 1. Гостевой доступ к каталогу кейсов

- `GET /api/v1/cases`, `GET /api/v1/cases/{id}`, `GET /api/v1/cases/featured` считаются публичными эндпоинтами.
- Во frontend маршруты `/cases` и `/cases/:id` доступны без авторизации.
- Действия, меняющие состояние, остаются под `auth:sanctum` (`POST /api/v1/cases/{id}/open`).

Причина: это уже отражено в API-правилах проекта и в текущих тестах публичного API.

## Решение 2. Безопасная эволюция callback пополнений

- Усиление callback включается конфигом (`SKINSARENA_DEPOSIT_CALLBACK_REQUIRE_SIGNATURE`), чтобы rollout был неразрушающим.
- Финальное целевое состояние для production: подпись callback обязательна.

Причина: сначала ввести проверку и тесты, затем включать enforcement по окружениям без аварийного отката.

## Решение 3. Ошибки API в едином формате

- Для `/api/*` все ошибки должны приводиться к формату:
  - успех: `{ "data": ... }`
  - ошибка: `{ "message": "...", "errors": {...} }`

Причина: единый контракт упрощает клиентскую обработку и снижает риск утечки внутренних деталей исключений.
